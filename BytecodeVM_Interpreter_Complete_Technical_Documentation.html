1. Executive Summary

BytecodeVM_Interpreter is a small educational compiler and virtual machine project written in C++. It implements a complete language execution pipeline: source text is tokenized by a lexer, parsed into an abstract syntax tree, compiled into bytecode instructions, and executed by a stack-based virtual machine.

The current program in main.cpp is hardcoded to compute factorial recursively: 

fact(5)

func fact(n) {
    if n == 0 {
        1
    }
    else {
        n * fact(n - 1)
    }
}

A validation build using g++ -std=c++17 produced: 

Result = 120

6. Token System

The token system is declared in token.h. A token is the smallest meaningful piece of source code. For example, the source x = 5 + 2 becomes: 

IDENTIFIER("x")
EQUAL("=")
NUMBER("5")
PLUS("+")
NUMBER("2")
EOF_TOKEN("")

6.1 TokenType Enum

enum TokenType {
    NUMBER,
    IDENTIFIER,

    PLUS,
    MINUS,
    STAR,
    SLASH,

    EQUAL,
    EQUAL_EQUAL,

    GREATER,
    LESS,

    LPAREN,
    RPAREN,

    LBRACE,
    RBRACE,

    IF,
    ELSE,
    WHILE,
    FUNC,
    COMMA,

    EOF_TOKEN
};

6.2 Token Class

class Token {
public:
    TokenType type;
    string value;

    Token() {}

    Token(TokenType t, string v) {
        type = t;
        value = v;
    }
};

The type tells the compiler what category the token belongs to. The value stores the exact text, such as "123", "+", or "fact".

7. Lexer

The lexer is declared in lexer.h and implemented in lexer.cpp. Its job is to scan the raw source string character by character and produce a vector of tokens.

7.1 Lexer State

class Lexer {
public:
    string text;
    int pos;
    char currentChar;

    Lexer(string input);
    void advance();
    vector<Token> tokenize();
    string identifier();
};
Member Meaning
text The complete source program.
pos Current character index.
currentChar The current character being scanned.

7.2 Lexer::advance

Moves one character forward. If the lexer reaches the end of the input, it stores '\0' as the end marker.

7.3 Lexer::identifier

Reads a sequence of letters, digits, or underscores. After reading the text, tokenize decides whether the text is a keyword such as if or a normal identifier such as total.

7.4 Lexer::tokenize

Main scanner loop. It recognizes numbers, identifiers, keywords, operators, parentheses, braces, and commas. Unknown characters are skipped by calling advance().

Compiler engineering note: silently skipping unknown characters is simple, but a production compiler should report lexical errors. For example, @ should probably produce an error instead of being ignored.

8. Abstract Syntax Tree

The AST represents the source program as structured objects. This is much easier to compile than raw tokens. All AST node classes are declared in ast.h.

AST Class Represents Important Fields
NumberNode An integer literal. value
BinaryOpNode Arithmetic operation. left, op, right
VariableNode Variable read. name
AssignNode Variable assignment. name, value
ProgramNode Whole program. statements
IfNode If/else statement. condition, ifBody, elseBody
CompareNode Comparison expression. left, op, right
WhileNode While loop. condition, body
FunctionDefNode Function definition. name, params, body
FunctionCallNode Function call. name, args

8.1 Example AST

Source:

2 + 3 * 4

AST shape:

BinaryOpNode("+")
├── NumberNode("2")
└── BinaryOpNode("*")
    ├── NumberNode("3")
    └── NumberNode("4")

This tree preserves precedence: multiplication is inside the right side of the addition.

9. Parser

The parser is a recursive descent parser. It consumes tokens and builds AST nodes. Its method structure encodes operator precedence.

9.1 Parser State

class Parser {
public:
    vector<Token> tokens;
    int pos;
    Token currentToken;

    Parser(vector<Token> t);
    void advance();
    AST* parse();
    AST* expr();
    AST* term();
    AST* factor();
    AST* statement();
    AST* program();
    AST* ifStatement();
    AST* comparison();
    AST* whileStatement();
    AST* functionDefinition();
    AST* functionCall();
};

9.2 Grammar Implemented

program        -> statement* EOF

statement      -> functionDefinition
                | ifStatement
                | whileStatement
                | assignment
                | comparison

assignment     -> IDENTIFIER "=" comparison

comparison     -> expr ((">" | "<" | "==") expr)*

expr           -> term (("+" | "-") term)*

term           -> factor (("*" | "/") factor)*

factor         -> NUMBER
                | IDENTIFIER
                | functionCall
                | "(" comparison ")"

ifStatement    -> "if" comparison "{" statement* "}"
                  ("else" "{" statement* "}")?

whileStatement -> "while" comparison "{" statement* "}"

functionDefinition
               -> "func" IDENTIFIER "(" params? ")" "{" statement* "}"

functionCall   -> IDENTIFIER "(" args? ")"

9.3 Operator Precedence

Precedence Parser Method Operators
Highest factor() Numbers, variables, function calls, parentheses
Middle term() *, /
Lower expr() +, -
Lowest comparison() >, <, ==

9.4 Parser Function Responsibilities

Function Responsibility
advance() Moves to the next token.
parse() Entry point; calls program().
program() Reads statements until EOF_TOKEN.
statement() Dispatches to function definition, if, while, assignment, or expression parsing.
comparison() Parses comparison expressions.
expr() Parses addition and subtraction.
term() Parses multiplication and division.
factor() Parses numbers, variables, function calls, and parenthesized expressions.
ifStatement() Parses condition, if body, and optional else body.
whileStatement() Parses loop condition and body.
functionDefinition() Parses function name, parameter list, and body.
functionCall() Parses function name and argument list.
Parser safety note: the parser assumes valid input. For example, it uses tokens[pos + 1] without a bounds check. A production parser should validate bounds and report structured syntax errors.

10. Bytecode And Instructions

Bytecode is a lower-level instruction sequence understood by the VM. Instead of directly interpreting the AST, this project compiles AST nodes into instructions such as OP_LOAD_CONST, OP_ADD, and OP_CALL.

10.1 Instruction Format

class Instruction {
public:
    OpCode opcode;
    string argument;

    Instruction(OpCode op, string arg = "") {
        opcode = op;
        argument = arg;
    }
};

10.2 Opcode Set

Opcode Meaning Stack Effect
OP_LOAD_CONST Push integer constant. [] -> [value]
OP_LOAD_NAME Push variable value from current frame. [] -> [locals[name]]
OP_STORE_NAME Pop value into current frame variable. [value] -> []
OP_ADD Add two integers. [a, b] -> [a + b]
OP_SUB Subtract two integers. [a, b] -> [a - b]
OP_MUL Multiply two integers. [a, b] -> [a * b]
OP_DIV Integer divide two integers. [a, b] -> [a / b]
OP_COMPARE_GT Greater-than comparison. [a, b] -> [a > b]
OP_COMPARE_LT Less-than comparison. [a, b] -> [a < b]
OP_COMPARE_EQ Equality comparison. [a, b] -> [a == b]
OP_JUMP_IF_FALSE Pop condition; jump if false. [condition] -> []
OP_JUMP Unconditional jump. No stack change.
OP_CALL Call a function by name. Arguments remain on stack until function prologue stores them.
OP_RETURN Return top stack value to caller. [returnValue] -> caller stack receives returnValue

11. Compiler

The compiler is declared in compiler.h and implemented in compiler.cpp. It converts AST nodes into bytecode instructions.

11.1 Compiler State

class Compiler {
public:
    vector<Instruction> instructions;
    map<string,int> functions;

    void compile(AST* node);
};
Member Meaning
instructions Generated bytecode program.
functions Maps function name to bytecode starting index.

11.2 Compilation Rules

AST Node Generated Bytecode
NumberNode OP_LOAD_CONST value
VariableNode OP_LOAD_NAME name
AssignNode Compile value, then OP_STORE_NAME name.
BinaryOpNode Compile left, compile right, then emit arithmetic opcode.
CompareNode Compile left, compile right, then emit comparison opcode.
IfNode Compile condition, conditional jump, if body, jump over else, else body.
WhileNode Mark loop start, compile condition, conditional exit jump, body, jump back.
FunctionDefNode Register function address, store parameters, compile body, emit OP_RETURN.
FunctionCallNode Compile each argument, then emit OP_CALL name.
ProgramNode Emit initial jump, compile function definitions first, patch jump, compile top-level statements.

11.3 Why ProgramNode Emits An Initial Jump

Function definitions are compiled into the same instruction stream as executable code. If execution started at instruction zero and the first instructions were a function body, the VM would accidentally execute the function without a call. To prevent that, the compiler emits: 

0: OP_JUMP main_start
1: function bytecode...
...
main_start:
N: top-level program bytecode...

This layout also lets top-level statements call functions that are defined later in source order.

12. Virtual Machine

The VM is declared in vm.h and implemented in vm.cpp. It executes the bytecode generated by the compiler.

12.1 VM State

class Frame {
public:
    map<string,int> locals;
    int returnAddress;
};

class VM {
public:
    vector<int> stack;
    vector<Frame> frames;
    map<string,int> functions;

    int getVariable(string name);
    void setVariable(string name, int value);
    int run(vector<Instruction>& code);
};
State Purpose
stack Evaluation stack for expression values and function return values.
frames Call stack. Each frame stores local variables and return address.
functions Function table copied from the compiler.

12.2 VM::run

run starts by creating a global frame with return address -1. Then it loops over bytecode using an instruction pointer: 

int ip = 0;

while (ip < code.size()) {
    Instruction instr = code[ip];
    switch (instr.opcode) {
        ...
    }
    ip++;
}

Jump and call instructions use continue after changing ip, because they do not want the normal ip++ to run.

12.3 Call Frames

A call frame represents one active function call. For the recursive call chain fact(5), the VM creates frames roughly like this: 

Global frame
fact(n = 5)
fact(n = 4)
fact(n = 3)
fact(n = 2)
fact(n = 1)
fact(n = 0)

When fact(0) returns 1, each older frame resumes, multiplies by its own n, and returns.

13. Entry Point

main.cpp connects the whole pipeline: 

Lexer lexer(input);
vector<Token> tokens = lexer.tokenize();

Parser parser(tokens);
AST* tree = parser.parse();

Compiler compiler;
compiler.compile(tree);

VM vm;
vm.functions = compiler.functions;

int result = vm.run(compiler.instructions);

The VM needs compiler.functions because OP_CALL stores only the function name. The VM uses the map to resolve the function name to an instruction index.

18. Function Calls And Important Parameter Limitation

Function arguments are compiled left to right: 

func sub(a, b) {
    a - b
}

sub(10, 3)

The call compiles arguments as:

OP_LOAD_CONST 10
OP_LOAD_CONST 3
OP_CALL sub

The stack before entering sub is: 

[10, 3]

The function prologue stores parameters using OP_STORE_NAME in parameter order: 

OP_STORE_NAME a
OP_STORE_NAME b

Since OP_STORE_NAME pops from the top of the stack, this binds: 

a = 3
b = 10

Therefore sub(10, 3) currently behaves like 3 - 10, returning -7.

Known bug: multi-argument function calls bind parameters in reverse order. A fix would be to store parameters in reverse order or compile arguments right to left.

19. Compiler And VM Concepts You Should Know

19.1 Lexer

A lexer converts characters into tokens. This simplifies parsing because the parser does not need to reason about individual characters. It sees meaningful categories such as NUMBER, IDENTIFIER, and PLUS.

19.2 Parser

A parser converts tokens into a structured representation. This project uses recursive descent, meaning grammar rules are implemented as C++ functions that call each other.

19.3 AST

An AST removes unnecessary syntax details and keeps the semantic structure. For example, parentheses influence the tree shape but are not usually stored as AST nodes.

19.4 Compiler

A compiler translates a high-level representation into a lower-level representation. Here, it translates AST into bytecode. In larger systems, a compiler might also perform semantic analysis, type checking, optimization, and machine-code generation.

19.5 Bytecode

Bytecode is an instruction format for a virtual machine. It is usually simpler than source code and more portable than native machine code. Examples include Java bytecode, Python bytecode, Lua bytecode, and WebAssembly-like instruction streams.

19.6 Stack Machine

A stack machine uses an implicit operand stack. Arithmetic instructions do not name registers. Instead, they pop operands from the stack and push results back. 

OP_LOAD_CONST 2   stack: [2]
OP_LOAD_CONST 3   stack: [2, 3]
OP_ADD            stack: [5]

19.7 Instruction Pointer

The instruction pointer, often called ip or program counter, stores the index of the next instruction to execute. Normal instructions increment it by one. Jumps and calls modify it directly.

19.8 Call Stack

The call stack stores active function calls. Each frame contains local variables and where to resume after the function returns.

19.9 Return Address

A return address is the instruction index after the call instruction. When a function returns, the VM sets ip back to this address.

20. Important Design Observations

20.1 Expression Statements Leave Values On Stack

If a statement is an expression, it leaves its value on the stack. This is why the last expression can become the program result. Assignment statements do not leave a value because OP_STORE_NAME pops the assigned value.

20.2 Undefined Variables Become Zero

getVariable uses: 

return frames.back().locals[name];

In std::map, using operator[] creates a missing key with a default value. For int, that default is 0. So reading an undefined variable silently gives 0.

20.3 Function Definitions Are Not First-Class Values

Functions are stored in the compiler and VM function table. They are not values that can be assigned to variables, passed as arguments, or returned from other functions.

20.4 No Lexical Closures

A function only accesses its current frame locals. It does not capture variables from the enclosing scope.

20.5 Runtime Type System Is Integer-Only

The VM stack is vector<int>. Comparisons push integer 0 or 1.

23. How To Extend The Project

23.1 Add A Return Keyword

Required changes:

  • Add RETURN token type.
  • Make lexer recognize return.
  • Add ReturnNode to ast.h.
  • Parse return statements in Parser::statement.
  • Compile ReturnNode to expression bytecode plus OP_RETURN.

23.2 Add Unary Minus

Modify factor() so if it sees MINUS, it parses the next factor and creates either: 

0 - expression

or a dedicated UnaryOpNode.

23.3 Add Error Reporting

Add line and column fields to tokens, then use parser helper functions like: 

expect(TokenType expected, string message)

This would let the parser say things like: Expected ')' after function arguments at line 3, column 14.

23.4 Fix Multi-Argument Function Calls

Option one: store function parameters in reverse order inside Compiler::compile(FunctionDefNode*). Option two: compile call arguments right to left. Option three: change OP_CALL to create a frame and explicitly move arguments into locals using known arity metadata.

25. Final Mental Model

To understand this project as a compiler engineer, keep this mental model: 

Lexer asks:
  "What are the words and symbols?"

Parser asks:
  "What structure do those words and symbols form?"

AST stores:
  "This is the program's tree-shaped meaning."

Compiler asks:
  "What VM instructions implement this tree?"

VM asks:
  "What should each instruction do to the stack, frames, variables, and instruction pointer?"

The project is small, but it contains the essential architecture used by many real interpreters: tokenization, parsing, AST construction, bytecode generation, stack execution, control flow, function calls, recursion, and call frames.
